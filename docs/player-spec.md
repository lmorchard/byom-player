# Technical Specification: `<byom-player>` Web Component

## 1. Project Overview

The `<byom-player>` is a framework-agnostic Web Component built with Lit and TypeScript. It serves as the frontend playback engine for JSPF-inspired JSON playlist manifests generated by the `byom-sync` CLI tool. It acts as a state machine, orchestrating UI and queue management while delegating actual audio playback to swappable "Audio Provider" adapters (e.g., streaming from a homelab Navidrome instance or falling back to a hidden YouTube iframe).

## 2. Tech Stack & Tooling

* **Core:** Lit (`lit` package), TypeScript
* **Build Tool:** Vite (configured to output a single, bundled standard ES module for easy dropping into static sites/Hugo)
* **Styling:** Encapsulated component CSS using Lit's `css` template literal, exposing CSS Variables for host-site theming (compatible with frameworks like Pico CSS).

## 3. Core Data Types

Define these interfaces in a central `types.ts` file to enforce the contract between the UI and the audio providers.

```typescript
export interface Track {
  title: string;
  artist: string;
  album?: string;
  isrc: string;
  duration_ms?: number;
  sync_state?: {
    spotify_present: boolean;
    date_orphaned?: string;
  };
}

export interface PlaylistManifest {
  title: string;
  creator?: string;
  date_created?: string;
  tracks: Track[];
}

export type ProviderState = 'uninitialized' | 'ready' | 'playing' | 'paused' | 'ended' | 'error';

export interface AudioProvider {
  name: string;
  initialize(): Promise<void>;
  load(track: Track): Promise<void>;
  play(): Promise<void>;
  pause(): void;
  seek(position_ms: number): void;
  onStateChange(callback: (state: ProviderState) => void): void;
}
```

## 4. Architecture: The Adapter Pattern

The player must completely decouple the UI from the audio source.

### A. The YouTube Provider (`providers/YouTubeProvider.ts`)

Acts as the universal fallback for public visitors.

* **Initialization:** Dynamically injects the YouTube IFrame Player API script (`https://www.youtube.com/iframe_api`). Creates a hidden `1x1` pixel `<iframe>` in the DOM.
* **Resolution:** When `load()` is called, it queries a configured API endpoint (or YouTube Data API directly if specified) using `"{artist} {title} audio"` to retrieve a `videoId`.
* **Playback:** Cues the `videoId` in the hidden iframe and maps the YT player state events to the generic `ProviderState` enum.

### B. The Direct Audio Provider (`providers/DirectProvider.ts`)

Acts as the zero-latency provider for homelab streaming (e.g., Navidrome on a Proxmox node).

* **Initialization:** Instantiates a standard HTML5 `new Audio()` object.
* **Resolution:** Expects the component environment or backend proxy to resolve the track's ISRC or metadata to a direct streamable URL (e.g., a `.flac` or `.mp3` stream).
* **Playback:** Manages the native `HTMLAudioElement` play/pause methods and event listeners.

## 5. The Lit Component (`ByomPlayer.ts`)

### Component API (Properties)

* `src` (String): The absolute or relative URL to the `.json` manifest file.
* `provider` (String, default: `"youtube"`): Determines which `AudioProvider` to instantiate on connection.

### Internal State (`@state()`)

* `manifest`: `PlaylistManifest | null`
* `currentTrackIndex`: `number` (default: 0)
* `playbackState`: `ProviderState`
* `progress`: `number` (0 to 100, tracking current playback position)

### Lifecycle & Behavior

1. **`connectedCallback`:** Fetch the JSON manifest from the `src` attribute. Instantiate the chosen `AudioProvider` and await its `initialize()` method.
2. **Queue Management:** When a track finishes (Provider emits `ended` state), automatically increment `currentTrackIndex`, call `provider.load()` with the new track, and call `provider.play()`.
3. **Error Handling:** If a track fails to resolve (Provider emits `error`), visually flag the track in the UI as unavailable and automatically skip to the next track.

### UI / Template Structure

The Shadow DOM should render:

1. **Header:** Playlist title and metadata.
2. **Now Playing:** The active track's title and artist.
3. **Controls:** Play/Pause button, Skip Forward, Skip Backward.
4. **Tracklist:** A scrollable `<ul>` or `<ol>` of tracks.
* The active track should receive an `.active` CSS class.
* Clicking a track item should immediately load and play that index.
* Orphaned tracks (`sync_state.spotify_present === false`) should have a distinct visual indicator (e.g., an icon or faded text) but remain clickable if the provider supports fallback resolution.
